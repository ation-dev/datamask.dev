Demo (V0)#

This page provides a conceptual, end-to-end walkthrough of how Datamask behaves in practice. It is illustrative, not a runnable distribution, and is intended to support design discussions and early feedback.

This demo focuses on behavior, boundaries, and determinism, not installation or execution.

What this demo demonstrates#

Declarative control#

All behavior is defined in configuration, not CLI flags or code.

Determinism#

Same input + same config → same output across runs.

Relationship safety#

Identifiers remain consistent across joins and derived datasets.

Demo flow#

flowchart TD
  A[Source<br/>Postgres / CSV] --> B[Config Validation]
  B --> C[Deterministic Transformation Engine]
  C --> D[Masked Output<br/>CSV / Non-prod DB]
  C --> E[Audit Log]

Demo artefacts#

CLI (illustrative)#

Datamask uses a thin CLI. All operational detail lives in configuration.

datamask run --config ./configs/demo-users.yml

Table-based configuration#

job:
  id: demo-users-v0
  mode: table

  rowSampling:
    enabled: true
    percentage: 20

  failFast: true

source:
  type: postgres
  connection:
    jdbcUrl: jdbc:postgresql://localhost:5432/demo_db
    username: demo_user
    password: demo_password
    schema: public

  table:
    name: users
    primaryKey: id

output:
  type: csv
  csv:
    path: ./out/users_masked.csv
    delimiter: ","
    includeHeader: true
  overwrite: true

audit:
  enabled: true
  format: jsonl
  path: ./out/audit.jsonl

strategies:
  user_id_token:
    type: deterministic_token
    params:
      namespace: users.id
      prefix: "U_"

  synthetic_full_name:
    type: synthetic_name
    params:
      locale: neutral

  synthetic_email:
    type: synthetic_email
    params:
      domain: example.test

  masked_phone:
    type: keep_last_n
    params:
      keep: 4
      maskChar: "*"

columns:
  users:
    id: { strategy: user_id_token }
    full_name: { strategy: synthetic_full_name }
    email: { strategy: synthetic_email }
    phone: { strategy: masked_phone }

Query-based configuration#

job:
  id: demo-orders-query-v0
  mode: query
  failFast: true

source:
  type: postgres
  connection:
    jdbcUrl: jdbc:postgresql://localhost:5432/demo_db
    username: demo_user
    password: demo_password

  query:
    sqlFile: ./queries/orders_enriched.sql
    params:
      country: "GB"
      min_amount: 1000
      from_date: "2025-10-01"
      to_date: "2025-12-31"

output:
  type: csv
  csv:
    path: ./out/orders_enriched_masked.csv
    includeHeader: true

audit:
  enabled: true
  path: ./out/audit.jsonl

strategies:
  user_id_token:
    type: deterministic_token
    params:
      namespace: users.id
      prefix: "U_"

  order_id_token:
    type: deterministic_token
    params:
      namespace: orders.id
      prefix: "O_"

  amount_perturb:
    type: numeric_perturb
    params:
      percent: 3
      clampMin: 0

columns:
  result:
    user_id: { strategy: user_id_token }
    order_id: { strategy: order_id_token }
    amount_minor: { strategy: amount_perturb }

Conceptual example#

Input (conceptual):

  • users.id = 1001
  • orders.user_id = 1001

Output (conceptual):

  • users.id = U_7f2a1c
  • orders.user_id = U_7f2a1c

The same identifier is transformed consistently wherever it appears.

Audit trail (illustrative)#

{"ts":"2026-01-06T21:12:03Z","job":"demo-users-v0","event":"job_start"}
{"ts":"2026-01-06T21:12:04Z","event":"config_validation","status":"ok"}
{"ts":"2026-01-06T21:12:05Z","event":"transform_summary","rows_read":3,"rows_written":3}
{"ts":"2026-01-06T21:12:05Z","event":"job_complete","status":"success"}

Audit logs are structured, append-only, and suitable for review or archival.
